# 框架的核心价值
1. 沉淀经验
2. 统一标准
3. 赋能团队 降低新成员(AI和人类)上手门槛。 一个新人无需再去通读几百页的开发规范,AI基于这个框架,自然地引导他遵循最佳实践

# 设计原则

1. 模块化 每一种AI能力(指令 技能 subagent专家) 都应该被封装在独立的、 可维护的文件或目录中。

2. 清晰地区分不同作用域的配置 如团队共享 vs 个人专属  项目级 vs 全局

3. 可共享  框架的核心部分应该能够被轻松地提交到Git仓库 随项目一起分发。

4. 可拓展 框架必须是 "活"的，能够随着团队能力提升以及经验的积累，方便地添加新的能力和规则


# AI原生开发框架全景

- 主要由两个大的部分构成: 
    - 项目级框架(团队共享资产)
        - 位于项目根目录下(./)  所有内容都应该提交到Git仓库  它定义了针对这个特定项目的、整个团队都必须遵守的规则和共享的能力
    - 用户级框架(个人专属配置)
        - 位于用户主目录下(~/)  所有内容都属于个人，不应该提交到任何项目仓库。它定义了跨所有项目的个人偏好和共享能力

- Claude Code 在启动时，会智能地将 这两个配置 进行合并与覆盖,最终形成一个独一无二、 既符合团队规范又尊重个人习惯的运行环境



- .claude/settings.local.json 这个文件非常关键,它允许团队在不修改共享的 settings.json的情况下,覆盖或添加自己私有的配置(如主题)


./.claude/commands
./.claude/skills
./.claude/agents
./.claude/hooks

./specs/:  规范驱动开发(SDD)
    - spec-kit 最佳实践是为每一个新特性(Feature) 或用户故事(User Story) 都在specs/ 下创建一个独立的目录,如 001-user-authentication/  这使得与单个特性相关的所有规范产物(spec.md plan.md tasks.md) 都被内聚地管理
    - 三大核心产物: 这个目录清晰地存放着我们从 意图 到 行动  的完整编译链, 是项目需求和技术方案的 活文档 和真理之源
